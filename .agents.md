# App

Zenith is a mathematical implementation of a productivity to-do app. It allocates
a daily time budget across tasks using the Zenith Gradient Algorithm.

**Math documentation map** (read before touching model code):

- `MATH.md` (repo root) — the AUTHORITATIVE record of the implemented math
  (model v2): every derivation, the v1→v2 change table with rationale, known
  approximations, and a doc-only revision log (§10) separating corrected
  explanations from actual model changes. Update it whenever the math
  changes; log doc-only corrections in §10.
- `zenith.md` (repo root) — frozen copy of the source article. Historical
  reference only; the implementation deliberately deviates from it (MATH.md §6
  lists how and why). Never "fix" code to match the article.
- `src/lib/business/model/zenith.test.ts` — executable spec: every identity
  (closed forms vs. numeric integration, root equations, allocator vs. brute
  force) is asserted there.

## Architecture

Three layers under `src/lib` with one-way imports (presentation → business →
data), enforced by eslint `no-restricted-imports`. See
`docs/code-conventions.md` for the rules.

## Files

- `src/lib/business/model/zenith.ts` — pure math (model v2): productivity
  curves, discrete block allocation with exact switch-cost subset search,
  dual-pool constraints, Bayesian personalization fit. Derivations in
  `MATH.md`.
- `src/lib/business/model/metric/calculation.ts` — task-level logic and
  dashboard metrics built on top of the zenith model.
- `src/lib/business/model/zenith-energy.ts` — experimental energy-stock model
  powering the `/energy` route ("Energy Lab"). Standalone by design: shares
  the curve/ϕ machinery with `zenith.ts` but none of its allocation code, so
  the main page is unaffected by changes here.
- `src/lib/business/store/session-store.svelte.ts` — the shared reactive
  daily session (tasks, budget, pools, flow logs + persistence).
- `src/lib/data/` — storage models, IndexedDB connection, repositories
  (`$`-prefixed CRUD controllers), localStorage→IndexedDB migration.

## Key model invariants (model v2 — full derivations in MATH.md)

- User inputs are 1–10; the model maps difficulty Eᵤ∈[1,10]→E∈[1,5] and
  enjoyment βᵤ∈[1,10]→β∈[1,2]. Metrics that compare E against β directly must
  account for this asymmetry (some deliberately use raw values instead).
- Productivity curve: p(t) = (a·kt + p₀)·e^(−kt) with k = (1−p₀/a)/ϕ, so
  p(0) = p₀ genuinely holds; peak at t = ϕ with value a·e^(p₀/a−1). The ratio
  r = p₀/a is capped at 0.9 (`AMPLITUDE_RATIO_CAP`).
- The single-task optimum is PER TASK: `T* = ϕ·x*(r)/(1−r)`, where `x*(r)`
  solves `eˣ = 1 + x + x²/(1+r)`; the multiplier ranges over (1.5, 1.7933].
  `OPTIMAL_PHI_MULTIPLIER` (1.7933) is only the r→0 limit / upper bound (and
  the energy model's seed). Use `TaskAllocation.optimalHours` for real values.
  The allocator never assigns time meaningfully past a task's `T*`.
- The objective is Σᵢ P̄ᵢ(tᵢ) — a sum of average productivity _rates_, not total
  output. P̄ jumps from 0 to ≈ p₀ at t = 0⁺ ("activation bonus"), so the
  objective is NOT concave — Lagrange/KKT-style solvers are invalid here. The
  allocator therefore works on discrete 15-minute blocks (`BLOCK_HOURS`):
  greedy marginal analysis (exact for the single budget), exhaustive
  funded-subset enumeration for switch costs (exact, n ≤ 12), plus a
  resource-transfer pass when a capacity pool binds (near-exact heuristic).
- Allocated hours are exact multiples of 0.25h; budget below one block is left
  unplanned. There is no 0.01h rounding step anymore.
- ϕ = c₁E + c₂β + c₃ is floored at 0.1h; constants are personalized via
  `fitUserConstants` — a Bayesian linear regression whose MAP equals the old
  ridge fit, plus posterior covariance/noise (`phiPredictionStd`) and an
  optional forgetting factor. The allocator consumes only the MAP.
- Allocations respect three constraints: the time budget, and cognitive/physical
  capacity pools (task weight = dimension difficulty / 10). Context switches
  cost `switchCost` hours (interpreted as attention residue — distinct from
  ramp-up, which ϕ already prices) and are only charged between tasks that
  receive time.

## Energy model invariants (`zenith-energy.ts`, /energy route only)

- Objective is Σ_tasks V(task's daily output), not Σ P̄ — total output per
  task pushed through the concave satiety wrapper V(O) = κ·ln(1+O/κ),
  κ = `satietyScale`·(that task's reference single-session output). Satiety
  breaks the old winner-take-all behavior (re-running the best task always
  beat switching); it must key on cumulative OUTPUT, never on the session
  phase, which decays over gaps and could be gamed with breaks.
  `satietyScale ≤ 0` recovers pure total output. The objective is only
  well-posed with its stopping terms: freeTimeValue (per hour not worked) and
  terminalEnergyValue (end-of-window energy). Fatigue alone never leaves the
  end of the window idle — it only produces instrumental mid-day rest.
  Derivation + rejected satiety forms: MATH.md §8.4.
- Warm-up p(s) uses a per-task session phase with DECAYING CARRYOVER: leaving
  a task for a gap g and returning resumes at s·e^(−g/τ)
  (`resumptionTimeConstant`), not 0 — brief switches cost little warm-up, long
  gaps approach a cold restart. `normalizeSchedule` merges adjacent same-task
  blocks. Fragmentation stays costly (probe-verified), just not the old
  hard-reset cliff.
- Reservoirs follow dC/dτ = −α·w·C + r'·g·(1−C) with recovery gate
  g = 1−(1−b)·w (b = `microRecoveryFraction`, default 0.05) and
  r' = recoveryRate·`restRecoveryMultiplier` (intermittent-rest correction) —
  closed-form exponential per block, no ODE solver. The gate keeps a
  full-demand (w = 1) task above the floor b·r'/(α+b·r') instead of draining
  to zero — without it, demand 10 vs 9.5 flips the plan (knife edge). A
  (1−w^q) gate does NOT fix this (still 0 at w = 1, probe-verified); don't
  re-propose it. b = 0 recovers the pure (1−w) gate. MATH.md §8.5.
  Output gate is Cobb-Douglas: C_cog^wc · C_phys^wp
  with demands w = dimensionDifficulty/10. Block output uses composite Simpson
  with ≥16 nodes per fastest timescale (min of ϕ, 1/ρ) — keeps relative error
  ~1e-6 even for near-floor ϕ tasks in long blocks.
- Optimizer is a deterministic multi-seed steepest-ascent local search over
  (task|rest, duration) block schedules — per the roadmap: not slot-greedy
  (myopic, never rests), not full DP. Pure single-step moves strand ~1% of
  objective and can return the wrong plan STRUCTURE, so it also has compound
  moves (transfer between blocks, half-block reassign, T*-session insert) and
  drop-one classic seeds — keep those when touching the search. MATH.md §8.6.
- The route never writes to the daily session; lab parameters persist in
  localStorage under `zenith-energy-params`.
