# App

Zenith is a mathematical implementation of a productivity to-do app. It allocates
a daily time budget across tasks using the Zenith Gradient Algorithm (see the
article link at the top of `src/lib/zenith.ts`).

## Files

- `src/lib/zenith.ts` — pure math: productivity curves, Lagrange/dual-pool
  allocation, switch-cost resolution, ridge personalization fit.
- `src/lib/metrics/calculations.ts` — task-level logic and dashboard metrics
  built on top of the zenith model.
- `src/lib/zenith-energy.ts` — experimental energy-stock model powering the
  `/energy` route ("Energy Lab"). Standalone by design: shares the curve/ϕ
  machinery with `zenith.ts` but none of its allocation code, so the main
  page is unaffected by changes here.

## Key model invariants

- User inputs are 1–10; the model maps difficulty Eᵤ∈[1,10]→E∈[1,5] and
  enjoyment βᵤ∈[1,10]→β∈[1,2]. Metrics that compare E against β directly must
  account for this asymmetry (some deliberately use raw values instead).
- Productivity curve: p(t) = (a+p₀)·k·t·e^(−kt) with k = 1/ϕ; peak at t = ϕ,
  single-task average optimum at t ≈ 1.7933ϕ (`OPTIMAL_PHI_MULTIPLIER`). The
  allocator never assigns time past a task's optimum.
- The objective is Σᵢ P̄ᵢ(tᵢ) — a sum of average productivity *rates*, not total
  output. P̄ is concave on [0, optimum], which is what makes the Lagrange /
  dual-pool solves globally valid.
- ϕ = c₁E + c₂β + c₃ is floored at 0.1h; the constants are personalized via a
  ridge fit (`fitUserConstants`) anchored to `DEFAULT_USER_CONSTANTS`.
- Allocations respect three constraints: the time budget, and cognitive/physical
  capacity pools (task weight = dimension difficulty / 10). Context switches
  cost `switchCost` hours and are only charged between tasks that receive time.
- Displayed hours are rounded to 0.01h; boundary comparisons against model
  quantities need a rounding epsilon.

## Energy model invariants (`zenith-energy.ts`, /energy route only)

- Objective is TOTAL OUTPUT (∫p·gate dτ), not Σ P̄. It is only well-posed with
  its stopping terms: freeTimeValue (per hour not worked) and
  terminalEnergyValue (end-of-window energy). Fatigue alone never leaves the
  end of the window idle — it only produces instrumental mid-day rest.
- Warm-up p(s) uses time-in-current-SESSION and resets on any switch or rest;
  `normalizeSchedule` merges adjacent same-task blocks into one session.
  Without the reset, confetti-slicing would be optimal (probe-verified).
- Reservoirs follow dC/dτ = −α·w·C + r·(1−w)·(1−C) — closed-form exponential
  per block, no ODE solver. Output gate is Cobb-Douglas: C_cog^wc · C_phys^wp
  with demands w = dimensionDifficulty/10. Block output uses composite Simpson
  with ≥16 nodes per fastest timescale (min of ϕ, 1/ρ) — keeps relative error
  ~1e-6 even for near-floor ϕ tasks in long blocks.
- Optimizer is a deterministic multi-seed steepest-ascent local search over
  (task|rest, duration) block schedules — per the roadmap: not slot-greedy
  (myopic, never rests), not full DP.
- The route never writes to the daily session; lab parameters persist in
  localStorage under `zenith-energy-params`.
